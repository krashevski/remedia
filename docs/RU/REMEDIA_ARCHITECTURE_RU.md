# АРХИТЕКТУРА REMEDIA

## 🧭 Обзор

Remedia — это модульная системная среда, построенная на основе идеи:
> **управляемого, наблюдаемого и восстанавливаемого состояния системы Linux**
Архитектура Remedia развивается не из пользовательского интерфейса, а из:
* среды выполнения
* контрактов
* отказоустойчивости
* воспроизводимости

## 🧱 Архитектура высокого уровня

```text
Пользователь
↓
CLI (remedia)
↓
Маршрутизатор (точка входа)
↓
Среда выполнения
↓
Реестр (модули + контракты)
↓
Модули (система / носитель / резервное копирование / демоверсия)
↓
Файловая система + ОС
```

## 🔀 Последовательность выполнения

```bash
remedia <модуль> <действие>
```

```text
→ CLI Маршрутизатор
→ Среда загрузка
→ Инициализация среды выполнения
→ Загрузка конфигурации (/etc/remedia/remedia.env)
→ Разрешение модулей (реестр)
→ Проверка контракта
→ Выполнение действия
→ Ведение журнала/обновление состояния
```

## 🧩 Основные уровни

### 1. Уровень CLI
**Точка входа**: `remedia`
Обязанности:
* анализ команд
* маршрутизация
* проверка среды
Ключевая идея:
> CLI — это тонкий слой, вся логика находится ниже

### 2. Уровень среды выполнения
Среда выполнения — это ядро ​​выполнения.

Обязанности:
* Изоляция среды (env -i)
* Загрузка системных переменных
* Управление пользователем (RUN_USER, HOME)
* Последовательности начальной загрузки
Особенности:
* Детерминированное выполнение
* Контролируемая среда
* Воспроизводимость

### 3. Уровень реестра
Реестр является **центром системных знаний**.

Содержит:
* Список модулей
* Метаданные
* Контракты
Обязанности:
* Обнаружение модулей
* Осведомленность о зависимостях
* Привязка контрактов

### 4. Модульная система
Модули — это независимые блоки логики.

```text
modules/
├── system/
├── mediasystem/
├── mediapanel/
├── backupkit/
└── demo/
```

Каждый модуль:
* имеет структуру
* реализует команды
* может иметь пользовательский интерфейс
* может иметь контракты

## 📦 Структура модуля (пример)

```text
module/
├── module.meta
├── commands/
├── ui/
├── lib/
└── contracts/
```

## 📜 Система контрактов

Контракты являются ключевым элементом Remedia.

Типы:
* предварительные условия
* постусловия
* инварианты
Используется для:
* проверки состояния перед действием
* обеспечения корректного выполнения
* предотвращения повреждения системы

## 🧠 Модель манифеста

Remedia представляет модель состояния, управляемую манифестом.
Манифест:
* описывает ожидаемое состояние системы
* хранится независимо от времени выполнения
* используется в качестве резервного варианта
Назначение:
* восстановление системы
* проверка целостности
* **якорь доверия** во время деградации

## 🔄 Транзакционная модель (в разработке)

Remedia переходит к транзакционному выполнению:
```text
BEGIN
→ validate
→ stage
→ execute
→ verify
COMMIT / ROLLBACK
```

Назначение:
* предсказуемые изменения системы
* безопасные операции
* возможность отката

## 🔍 Подсистема Doctor (планируется)

Doctor — это глобальный диагностический слой. Проверки:
* dpkg
* файловая система
* использование диска
* GPU
* пользовательская среда
Результаты:
* СОСТОЯНИЕ СИСТЕМЫ: XX%

## 💾 Стратегия файловой системы

Remedia использует явную модель путей:
* FAST_STORAGE → /mnt/shotcut
* SLOW_STORAGE → /mnt/storage
* BACKUP_STORAGE → /mnt/backups
* PROJECT_DIR → /mnt/storage/Videos/projects
Особенности:
* разделение по скорости и назначению
* мониторинг использования
* осведомленность на системном уровне

## 🎬 Архитектура медиаконвейера

MediaSystem реализует конвейер:
```text
INGEST
→ COPY
→ VERIFY
→ CLEAN

PROCESS
→ PROXY
→ AUDIO CLEAN
→ SYNC

POST
→ SPLIT
```

Особенности:
* Идемпотентность
* Возможность возобновления
* Ведение журнала каждого шага

## 🖥 Панель управления медиафайлами (интерфейс командной строки)

Панель управления медиафайлами — это уровень оркестрации.

Позволяет:
* Управлять проектами
* Запускать конвейеры
* Отслеживать состояние
* Видеть всю систему
Важно:
> Пользовательский интерфейс — это слой управления, а не источник логики

## ♻️ Архитектура восстановления (BackupKit)

BackupKit предоставляет:
* Подход с использованием моментальных снимков
* Восстановление пользовательских данных
* Восстановление среды
Философия:
> Система должна быть способна к восстановлению

## 🔐 Принципы безопасности

Remedia построена на основе:
* Безопасного выполнения
* Явного состояния
* Предварительной проверки
* Минимального доверия к внешним пакетам

## ⚙️ Конфигурация

Основной файл конфигурации:
```bash
/etc/remedia/remedia.env
```

Содержит:
* Пути
* Режимы работы
* Параметры среды

## 🧭 Принципы проектирования

### 1. Восстановление в первую очередь
Система должна быть Способность выдерживать сбои.

### 2. Явное состояние
Никакой «магии», всё описано.

### 3. Детерминированная среда выполнения
Один и тот же ввод → один и тот же результат.

### 4. Модульная архитектура
Компоненты независимы.

### 5. CLI-ориентированность
Пользовательский интерфейс второстепенн.

## 🚧 Будущая архитектура

* Выполнение в песочнице (предварительная установка пакетов)
* Полноценный транзакционный движок
* Распределенные манифесты
* Конвейеры с поддержкой GPU (NVENC)
* Экосистема плагинов
