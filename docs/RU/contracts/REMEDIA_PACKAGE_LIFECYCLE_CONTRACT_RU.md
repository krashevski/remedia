# 📦 СПЕЦИФИКАЦИЯ ЖИЗНЕННОГО ЦИКЛА ПАКЕТА REMEDIA v1.0

## 🧠 Основная философия

> Разделение системы на неизменяемый код, изменяемую конфигурацию и постоянное состояние.

```txt
| Layer              | Meaning                | Mutability         |
| ------------------ | ---------------------- | ------------------ |
| `/usr/lib/remedia` | code (engine, modules) | ❌ immutable        |
| `/etc/remedia`     | configuration          | ⚠️ user-controlled |
| `/var/lib/remedia` | runtime state          | ✔ mutable          |
| `/var/log/remedia` | logs                   | ✔ append-only      |
| `/run/remedia`     | ephemeral runtime      | ✔ volatile         |
```

## 🔁 1. ЖИЗНЕННЫЙ ЦИКЛ УСТАНОВКИ

### 🧩 1.1 preinst (необязательный защитный механизм)
**Назначение:**
* Обеспечить безопасность установки/обновления системы.
**Разрешено:**
* проверить совместимость ОС
* остановить старый демон (если существует)
* создать резервную копию критического состояния среды выполнения (необязательно)
**Запрещено:**
* записать конфигурацию
* инициализировать среду выполнения
* запустить службы

### 📦 1.2 фаза распаковки (dpkg)
**Ответственность dpkg:**
* распаковать `/usr/lib/remedia`
* распаковать `/etc/remedia` (если предоставлено)
* зарегистрировать конфигурационные файлы

### ⚙️ 1.3 postinst (фаза загрузки)
**Назначение:**
Сделать систему работоспособной, а не операционно-функциональной.
**Разрешено:**
* создание отсутствующих каталогов
* обеспечение существования конфигурации (ТОЛЬКО если она отсутствует)
* установка разрешений
* инициализация пустых каталогов среды выполнения
* запуск загрузки кэша/индекса (только безопасный идемпотентный подход)
**Запрещено:**
* выполнение модулей
* выполнение команд CLI
* запуск полного механизма среды выполнения
* изменение существующей конфигурации, если отсутствуют ключи
**Модель Postinst:**
```bash
POSTINST = "сделать систему работоспособной, а не активной"
```

### 🧪 1.4 Первая загрузка (ленивая инициализация среды выполнения)
**Триггер:**
* первое выполнение исправления
**Ответственность:**
* полная инициализация среды выполнения
* загрузка реестра
* активация модуля
👉 ЭТО заменяет сложную логику postinst

## 🔁 2. ЖИЗНЕННЫЙ ЦИКЛ ОБНОВЛЕНИЯ

### 📦 2.1 Распаковка новой версии
dpkg перезаписывает `/usr/lib/remedia`
### ⚠️ 2.2 Обработка конфигурационных файлов (критически важно)
**Правило:**
> `/etc/remedia/*` НИКОГДА не перезаписывается автоматически
Если изменено:
* dpkg запрашивает у пользователя подтверждение
* или сохраняет .dpkg-old
### 🔧 2.3 postinst (режим обновления)
**Должен быть идемпотентным**
Разрешено:
* обнаружение изменения версии
* запуск скриптов миграции
* обновление отсутствующих ключей конфигурации
* перестройка кэша
* обеспечение совместимости
Запрещено:
* сброс конфигурации
* удаление состояния
* сброс пользовательских данных
### 🧬 Контракт обновления:
```txt
Upgrade = "развивать систему, никогда не сбрасывать ее"
```

### 🧩 2.4 Слой миграции (ОЧЕНЬ ВАЖНО!
**Расположение:**
```bash
/usr/lib/remedia/migrations/
```

**Структура:**
```bash
migrations/
v1_to_v2.sh
v2_to_v3.sh
```

**Правила:**
* должен быть безопасен для повторного запуска
* должен учитывать версию
* никогда не должен удалять конфигурацию пользователя вслепую

## 3. УДАЛЕНИЕ ЖИЗНЕННОГО ЦИКЛА

### 🧪 3.1 Удаление (сохранение данных)
**Разрешено:**
* остановить среду выполнения
* удалить бинарные файлы
* сохранить `/etc` и `/va`r (поведение Debian по умолчанию)
**Результат:**
систему можно переустановить без потери данных
### 💣 3.2 очистка (полная очистка)
**Только если пользователь явно запросит:**
```bash
apt purge remedia
```

**удаляет:**
```bash
/etc/remedia
/var/log/remedia
```

## ⚡ 4. МОДЕЛЬ ЗАПУСКА ВРЕМЕНИ ВЫПОЛНЕНИЯ (подобная systemd)

### ❌ Нет автозапуска в postinst
### ✔ Модель запуска:
```bash
user runs: remedia system center
→ runtime_init
→ registry load
→ module execution
```

## 🧠 5. МОДЕЛЬ ЭВОЛЮЦИИ КОНФИГУРАЦИИ

**Правило:**
> Конфигурация НИКОГДА не перезаписывается, только расширяется.
**Безопасная стратегия слияния:**
```text
if key missing → add default
if key exists → preserve
if key invalid → warn only
```

**Пример:**
```bash
FAST_STORAGE=/mnt/shotcut
SLOW_STORAGE=/mnt/storage
BACKUP_STORAGE=/mnt/backups
LOG_DIR=/var/log/remedia
```

## 🧱 6. МОДЕЛЬ СОСТОЯНИЯ (подобная Docker)

```bash
/var/lib/remedia
```

Должна вести себя как:
> слой состояния контейнера
**Правила:**
* постоянное состояние
* безопасное обновление
* никогда не затрагивается postinst, кроме mkdir
* контролируется миграцией

## 🔐 7. ГАРАНТИИ БЕЗОПАСНОСТИ

**После установки или обновления:**
✔ Должен запускаться CLI
✔ Должна существовать конфигурация
✔ не требуется ручное восстановление
✔ нет поврежденного состояния
✔ система всегда загружается

## 🚫 8. ЗАПРЕЩЕННОЕ ПОВЕДЕНИЕ

**НИКОГДА:**
* запускайте модули в postinst
* инициализируйте среду выполнения в скриптах установки
* перезаписывайте `/etc/remedia/remedia.env`
* удаляйте `/var/lib/remedia` во время обновления
* предполагайте, что пользовательская среда существует

## 🧭 9. КОНЕЧНЫЙ АВТОМАТ СИСТЕМЫ

```bash
INSTALL:
  unpack → postinst → ready → first-run init

UPGRADE:
  unpack → conffile preserve → migrations → ready

REMOVE:
  stop → uninstall binaries → keep data

PURGE:
  full wipe
```

## 🧠 10. ЗАКЛЮЧИТЕЛЬНАЯ МЕНТАЛЬНАЯ МОДЕЛЬ

🐳 **Docker + systemd гибрид**
```txt
| Concept         | Equivalent       |
| --------------- | ---------------- |
| image           | /usr/lib/remedia |
| container state | /var/lib/remedia |
| config volume   | /etc/remedia     |
| logs            | /var/log/remedia |

```
