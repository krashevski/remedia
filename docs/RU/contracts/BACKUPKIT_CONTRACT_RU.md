# 🧠 BACKUPKIT v3 — Спецификация производственной архитектуры

🎯 Основная идея
Разделить систему на три независимых временных уровня:
1. РЕАЛЬНОЕ ВРЕМЯ (Резервное копирование / Восстановление)
2. ОФФЛАЙН (Проверка / Аудит)
3. ЭКСПОРТ (Холодный архив)
И исключить все ресурсоёмкие операции из среды выполнения.

## 🧱 1. ОСНОВНЫЕ УРОВНИ

🟢 УРОВЕНЬ 1 — МЕХАНИЗМ РЕЗЕРВНОГО КОПИРОВАНИЯ (БЫСТРЫЙ ПУТЬ В РЕАЛЬНОМ ВРЕМЕНИ)
📌 Назначение
Создать снимок состояния системы.

📦 Вывод:
```bash
/snapshots/<id>/
```

📌 Контракт
```text
INPUT:
src: /home/$USER

OUTPUT:
snapshot/
optional: archive.tar.zst (cold export trigger only)
```
ПРАВИЛА:
- ЗАПРЕЩЕНО хеширование
- ЗАПРЕЩЕНО создание манифестов
- ЗАПРЕЩЕНО сравнение изменений
- ЗАПРЕЩЕНА проверка, выходящая за рамки проверки целостности rsync

⚙️ Разрешенные операции:
* создание снимка rsync
* копирование файла
* запись метаданных (meta.json)
❌ Запрещено:
* сканирование sha256
* анализ обхода файловой системы
* сравнение снимков

## 🟡 УРОВЕНЬ 2 — ДВИЖОК ВОССТАНОВЛЕНИЯ (БЕЗОПАСНЫЙ ПУТЬ СЛИЯНИЯ)

📌 Назначение
Восстановление данных без уничтожения текущего состояния. 📦 Режимы:
```text
snapshot
no-regression
hybrid
```

📌 Контракт
```bash
INPUT: 
snapshot/ 
optional archive/

OUTPUT: 
restored filesystem state
```

ПРАВИЛА:
- НИКОГДА не перезаписывать более новые файлы
- НИКОГДА не удалять без явного слоя удаления

🔒 Основное правило:
> «цель всегда является авторитетной, если она новее»
🧠 Разрешенные инструменты:
- rsync --update (безопасное слияние)
- tar --keep-newer-files
- опциональный слой удаления (только явное удаление)
❌ Запрещено:
- использование манифеста
- пересчет целостности
- сканирование всей файловой системы

## 🔵 УРОВЕНЬ 3 — ДВИГАТЕЛЬ ДОКТОРА (ОФФЛАЙН-АНАЛИТИКА)

📌 Назначение
Анализ состояния системы после факт.

📦Выходные данные:
```bash
manifest.txt
diff report
integrity report
fs audit report
```

📌 Контракт
```bash
INPUT: 
snapshot A 
snapshot B (optional)

OUTPUT: 
structured report (JSON)
```

ПРАВИЛА:
- Может быть медленным
- Может сканировать всю файловую систему
- Может вычислять хеши
- НЕ ДОЛЖЕН влиять на работу системы во время выполнения

🧠Особенности:
* Полное сканирование SHA256
* Сравнение снимков
* Обнаружение удаленных файлов
* Обнаружение аномалий (необязательный будущий уровень)
🔥Варианты использования:
* Анализ инцидентов безопасности
* Регрессионная отладка
* Обнаружение системных отклонений
* Судебно-криминалистическое расследование

## 🟣 УРОВЕНЬ 4 – ДВИЖОК ХОЛОДНОГО АРХИВИРОВАНИЯ (ЭКСПОРТ) СЛОИ)

📌Назначение
Длительное хранение, портативность.

📦 Вывод:
```bash
archive.tar.zst
```

📌Контракт:
```bash
INPUT: 
snapshot/

OUTPUT: 
compressed archive
```

ПРАВИЛА:
- нет зависимостей от среды выполнения
- нет интроспекции файловой системы

## 🔄 МОДЕЛЬ ПОТОКА СИСТЕМЫ

🟢 Поток резервного копирования
```bash
backupkit backup
├── rsync snapshot (быстрый)
├── write meta.json
└── optional cold archive
```

🟡 Поток восстановления
```bash
backupkit restore 
├── snapshot restore (rsync safe merge) 
├── optional archive patch 
└──NO manifest involvement
```

🔵 Процесс проверки подлинности
```bash
backupkit doctor 
├── generate manifest 
├── compare snapshots 
├── verify integrity 
└── produce report
```

## 🚫 ЖЕСТКИЕ АРХИТЕКТУРНЫЕ ПРАВИЛА

❌ НИКОГДА НЕ ДЕЛАЙТЕ:
- манифест в конвейере резервного копирования
- SHA256 при восстановлении
- межслойные зависимости
- сканирование файловой системы во время выполнения
- блокирующие операции хеширования
✅ ВСЕГДА ДЕЛАЙТЕ:
- резервное копирование = быстро
- восстановление = безопасно
- проверка = медленно, но эффективно
- архивирование = опциональный экспорт

## 🧠 ФИЛОСОФИЯ ПРОЕКТИРОВАНИЯ

Основной принцип:
> «Быстрый путь никогда не должен зависеть от медленного пути».

Модель разделения времени:
```text
| Layer | Time cost | Purpose |
| ------- | --------- | ---------- |
| Backup | fast | snapshot |
| Restore | medium | safe merge |
| Doctor | slow | analysis |
| Archive | optional | storage |
```
